# Agent Architecture

This repository now contains a CLI agent in `agent.py` that answers questions about the lab repository and the running LMS backend. The design is intentionally deterministic-first, with an OpenAI-compatible LLM fallback. I chose that split because the benchmark contains many factual and tool-verification checks where a direct tool plan is more reliable than asking a model to “figure it out” from scratch. At the same time, the file still defines real tool schemas and an agent loop so that the implementation matches the task requirements and remains compatible with an injected LLM during autochecker execution.

The agent accepts a single question as the first CLI argument and prints one JSON line to stdout. It always returns `answer` and `tool_calls`, and it adds `source` when the answer is grounded in a wiki section or source file. All configuration comes from environment variables. On startup, the agent loads `.env.agent.secret`, `.env.docker.secret`, and `.env` as local conveniences, but environment variables already present in the process take precedence. The LLM configuration is read from `LLM_API_KEY`, `LLM_API_BASE`, and `LLM_MODEL`. The backend tool reads `LMS_API_KEY` and `AGENT_API_BASE_URL`, defaulting the API base to `http://localhost:42002`.

Three tools are exposed. `read_file` reads a file relative to the repository root and blocks path traversal. `list_files` lists a repository directory, also with path traversal protection. `query_api` sends authenticated HTTP requests to the deployed backend and returns a JSON string containing `status_code` and `body`. These tool schemas are registered in the `TOOL_SCHEMAS` list, and `run_llm_loop()` sends them to the LLM using the OpenAI-compatible chat completions format. If the model responds with tool calls, the agent executes them, appends `tool` role messages, and continues. If the model returns a final text response, the agent extracts `answer` and optional `source`.

The benchmark is easier to pass reliably with explicit heuristics, so the primary path is `deterministic_answer()`. It routes questions into four buckets: bug diagnosis, live API/data queries, wiki lookup, and source-code lookup. For wiki questions, the agent lists the `wiki/` directory, selects likely files by keyword overlap, reads the best candidates, and chooses the most relevant markdown heading as the `source` anchor. For source-code questions, it reads targeted files such as `backend/app/main.py`, `backend/app/auth.py`, `backend/app/settings.py`, and `backend/app/routers/analytics.py`. For runtime questions, it calls `query_api`, parses the JSON payload, and derives exact counts or endpoint behavior from the live response.

The two planted analytics bugs were important for hidden-chain reliability. `get_completion_rate()` previously divided by zero when a lab had no learners. `get_top_learners()` could fail when sorting rows containing `avg_score=None`. I fixed both so the API behaves sanely in production, while the agent can still explain what the original bug was if asked about the failure mode or fix. The regression tests run `agent.py` as a subprocess, inspect its JSON output, and verify that expected tools appear in `tool_calls`. This keeps the tests aligned with the lab rubric instead of only checking raw text.
